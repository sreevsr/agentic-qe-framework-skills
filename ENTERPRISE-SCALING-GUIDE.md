# Enterprise Scaling Guide

> How to use the Agentic QE Framework for enterprise projects.

---

## The Problem

When multiple teams or applications want to use this framework, you have three options. Two of them are wrong.

**Wrong option 1** — Single monolithic repo for all apps. Each app has different baseURLs, auth, timeouts, and browser needs. One team's broken test blocks another team's CI. Merge conflicts become daily friction. Pages and locators have zero overlap across apps, so nothing is truly "shared."

**Wrong option 2** — Fully independent copies with no connection. When you improve an agent prompt or fix base-page.ts, you manually propagate to every copy. Over time the copies drift and you lose framework consistency.

**The Right Pattern**: Template Repo + Application Repos
Maintain two tiers:
Tier 1 — Framework Template Repo (maintained by QCoE / framework owner)
This is the reusable engine. It contains agents, templates, core utilities, and prompt files. It never contains application-specific artifacts.

**How Framework Updates Flow**
When you improve an agent prompt, fix base-page.ts, or add a new keyword:

1. Push the change to the template repo (Tier 1)
2. Each team pulls the update into their application repo
3. Only agents/ and templates/ folders change
4. Team scenarios, output, and application-specific code remain untouched

This is the same pattern used by Yeoman generators, Create React App,
and Spring Initializr — a shared scaffolding tool that produces
independent, self-contained projects.

**When to Use Which**

Situation                   |               Approach                        |
Single team, single app     |   Use template repo directly, add scenarios   |
Multiple teams, different apps  |   One template repo + one application repo per team   |
Same app, multiple modules  |   Single application repo, one scenario .md per module    |
Customer demo / POC         |   Use template repo with sample scenarios included    |


**Governance**
The template repo is owned by QCoE / framework owner. Application teams can write scenarios but cannot modify agents or templates without review. 
Quality standards flow through the agent prompts — the Reviewer agent
enforces the same 8-dimension audit regardless of which team runs it.


**When Prompts Get Too Long, Split the Scenario Instead**

If a scenario has 25+ steps, break it into smaller .md files using
CAPTURE/SAVE for data handoff. Two 15-step scenarios will produce more
reliable results than one 30-step scenario — for both the Generator
and the LLM's attention span.

This is the same principle as the framework itself: the reason 5 focused
agents outperform 1 monolithic agent is the same reason 3 focused scenarios
outperform 1 sprawling scenario. Keep each unit small enough that the LLM
can hold the entire context without dropping anything.
